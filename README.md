## memo: structure

FRPを作りたい、、

FRPの粘土をこねくり回すような感覚が好き、、インターフェースもそんなふうにできたらなと思うけど。。

でも、ECSのサッパリ感も好きだ。。グローバル変数として全部が全部とつながっているという、、うーん

FRPの基本単位はscan関数と言える。モナドではa -> m aの形をした関数を用いて通常のデータ構造変換関数からモナドへの埋め込みを行っているのと同様に、scan関数ではモナドからarrowへの埋め込みを行うことができる、、たぶんそんな感じ。。

まあrustだとlifetimeというとても厄介な問題があるわけだが、そもそもscanはlifetimeとかなり相性が悪い気がする。。各scanには内部変数が存在していて、それこそがscanの本質でもある。。その内部変数がいつまで使われるのかは不明瞭だ。。

そもそもグローバル変数は普通はごちゃごちゃするものだが（そのごちゃごちゃ感がいいのではないかという話はおいておいて、）ECSでは変数のスコープには制約を加えないものの、read/writeする関数の方に非常に厳しい制約を与えている（すなわち、変数も関数も両方fragmentedにするということ。。）から成り立つというのはある。。。。scanの内部状態をグローバル変数として捉えた場合それは必然的に階層化されることになり、上記のようなfragmentedな制約は、関数にも変数にも成り立たないように見える。。（それは必ずしも悪いことではないのかもしれないが。。）そしてECSはscanと違ってlifetimeとの相性は抜群と言える。そもそもcomponent/entityの着脱生成消滅がシステムの根幹にあるため、そこを実装すれば自動的にメモリ管理も実装されることになる。lifetimeとかいう概念自体（ECSの枠組みの中で限れば）一切必要ない。もちろん、各フレーム内の各関数の実行過程においてrustのlifetime計算によるメモリ管理は利用されるが、そこを出てしまえばECSの管轄になり、lifetimeの出る幕はない。。
